# 性能トレンド分析 ガイド

## 1. 概要
性能トレンド分析ダッシュボードは、ウェブアプリケーションの長期的な性能推移を可視化し、システムの状態を多角的に分析するためのツールです。1年間のデータを1時間単位の精度で保持し、トレンドの把握やキャパシティプランニングに活用できます。

## 2. メイン時系列チャート
アプリケーション上部に配置されたメインチャートは、選択したメトリクスの時間経過に伴う変化を確認するための主要な指標です。
1年間のデータを1日単位の精度で保持し、トレンドの把握やキャパシティプランニングに活用できます。

### 2.1 各指標（メトリクス）の定義
- **応答時間 Response Time (ms):** リクエストの処理完了までにかかった平均時間。ユーザー体験に直結する重要な指標です。
- **TPS Transactions Per Second:** 1秒あたりの平均処理件数。システムの現在のスループットを示します。
- **Hit数 Hits (Count):** 特定の時間枠内の総リクエスト数。システムの負荷量（トラフィック）を測定します。
- **同時ユーザ数 Concurrent Users:** システムにアクティブに接続しているユーザー数。
- **エラー率 Error Rate (%):** 総リクエスト数に対するエラー発生数の割合。計算式: `(エラー件数 / 総リクエスト数) × 100`。エラー率の上昇は、アプリケーションの不具合やインフラ障害を示す重要なシグナルです。
- **CPU / メモリ:** インスタンスごとの物理リソース使用率。平均値と最大値(Peak)を同時に分析することで、リソース不足を早期に検知できます。

### 2.2 ダッシュボードの操作方法
- **期間の絞り込み:** メインチャート上の特定の範囲をマウスでドラッグすると、その期間の統計データが全カードとヒートマップに即座に反映されます。キーボードの方向キー (←, →) を押すと、選択した1ヶ月の範囲を前後に範囲が拡張されます。
- **範囲の移動:** キーボードの方向キー (←, →) を押すと、選択した1ヶ月の範囲を前後に移動できます。
- **選択解除:** `Esc` キーを押すと、選択がリセットされ、1年全体のデータに戻ります。
- **移動平均の活用:** 「移動平均」をOnにすると、短期的な変動を抑えた長期的なトレンドが視覚化され、本質的な変化を捉えやすくなります。

### 2.3 なぜこのチャートが重要か？
- **長期トレンドの可視化:** 1年分のデータを1日単位で表示することで、月次・四半期ごとの性能推移や季節性のあるトラフィック変動を把握できます。一時的なスパイクと構造的な劣化を明確に区別できます。
- **キャパシティプランニング:** 過去の推移から将来のリソース需要を予測し、「いつ頃インフラ増設が必要か」を定量的に判断する根拠となります。
- **変更影響の検証:** デプロイやインフラ変更の前後で性能がどのように変化したかを時系列で確認でき、変更の効果やリグレッションを即座に検知できます。

### 2.4 移動平均線の活用方法
移動平均(MA)は日次の変動ノイズを除去し、本質的なトレンドを浮き彫りにします。本ツールでは **7日移動平均** と **30日移動平均** の2種類を提供しています。

#### ▶ 7日移動平均 (短期トレンド)
- 週単位の傾向変化を素早く検出できます。
- デプロイ後の性能変化を数日以内に確認する際に有効です。

#### ▶ 30日移動平均 (長期トレンド)
- 月単位の構造的なトレンドを把握できます。
- 30日線が持続的に上昇（応答時間・エラー率の場合）している場合、スケーリングやチューニングの検討が必要です。

#### ▶ 分析のポイント
- **7日線と30日線の乖離:** 7日線が30日線を上回り始めたら直近の性能悪化の兆候、下回り始めたら改善の兆候です。
- **ゴールデンクロス / デッドクロス:** 短期線が長期線を下から上に突き抜けた場合（応答時間等では悪化シグナル）、逆に上から下に抜けた場合は改善シグナルとして読み取れます。
- **原データとの差:** 原データが移動平均線から大きく乖離した日は、一時的なイベントや障害を示す外れ値である可能性が高いです。

## 3. Overall ヒットマップ分析 (相関分析)
このチャートは、**負荷(Hits)**と**性能指標**の相関を示します。データは中央値(Median)によって4つのエリアに分類されます。
性能指標の中央値(Median)は全体期間の中央値(Median)を使用します。Hit数の中央値(Median)は選択した期間の中央値(Median)を使用します。

- **左上: 低負荷 × 高指標値** 負荷(Hits)が少ないにもかかわらず、Y軸の指標値が高い状態。
- **右上: 高負荷 × 高指標値** 負荷(Hits)と Y軸の指標値が共に高い状態。
- **左下: 低負荷 × 低指標値** 負荷(Hits)と Y軸の指標値が共に低い状態。
- **右下: 高負荷 × 低指標値** 負荷(Hits)が高いにもかかわらず、Y軸の指標値が低い状態。

### 3.1 メトリクス別の解析ガイド
Y軸に表示される指標の種類によって、各エリアの意味が変わります。

#### ▶ 応答時間 / エラー率 / CPU / メモリ (値が高い＝悪化)
- **左上（⚠️）:** 低負荷でも指標が高い → アプリケーション内部の非効率（遅いクエリ、メモリリーク等）が疑われます。
- **右上（🔴）:** 高負荷で指標も高い → キャパシティの限界。リソースの増設やチューニングが急務です。
- **左下（🟢）:** 低負荷で指標も低い → 安定した理想的な状態です。
- **右下（✅）:** 高負荷でも指標が低い → 高効率。十分なキャパシティがあります。

#### ▶ TPS (値が高い＝良好)
- **左上（✅）:** 低負荷で高スループット → 効率的な処理が行われています。
- **右上（🟢）:** 高負荷で高スループット → 最適なパフォーマンス状態です。
- **左下（⚪）:** 低負荷で低スループット → アイドル状態またはトラフィックが少ない時間帯です。
- **右下（⚠️）:** 高負荷で低スループット → ボトルネックの可能性。処理が追いついていません。

### 3.2 なぜこのチャートが重要か？
- **負荷と性能の因果関係の把握:** 時系列チャートだけでは「いつ遅くなったか」は分かりますが、「なぜ遅くなったか」は分かりません。Overallヒートマップは、負荷量との相関を可視化することで原因の切り分けを支援します。
- **キャパシティの限界点の特定:** データポイントが右上エリアに集中し始めた場合、現在のインフラ構成ではトラフィックの増加に耐えられないことを示しており、事前のスケーリング計画に役立ちます。
- **チューニング効果の検証:** アプリケーションやインフラのチューニング前後でデータ分布がどう変化したかを視覚的に比較できます。

### 3.3 活用ガイド
- **分布の偏りを確認:** 左下（安定）に密集していれば健全な状態。右上へ広がるほどキャパシティリスクが高まっています。
- **期間絞り込みとの併用:** メインチャートで特定の月を選択すると、その期間だけの負荷-性能分布を確認でき、月別・季節別の変化を比較分析できます。
- **メトリクス切替による多面的分析:** 同じ期間でメトリクスを切り替えることで、「負荷増加時に応答時間は悪化するがエラー率は安定」など、複合的な性能特性を把握できます。
- **インスタンス別比較:** インスタンスを切り替えて分布パターンを比較することで、特定のサーバーだけに問題が集中していないか確認できます。

## 4. 曜日 x 時間帯別 分析 (パターン分析)
曜日と時間帯ごとの平均値をヒートマップで可視化し、システムの稼働パターンを一目で把握できるチャートです。統計的な四分位範囲(IQR)に基づき、自動的にステータスを分類します。

### 4.1 ステータスの定義
- **安定 (Stable):** 過去の統計から見て平均的な動作範囲（Q3以下）。
- **警戒 (Warning):** 通常よりも高い値を示す時間帯（Q3 〜 Q3+1.5×IQR）。注意深い監視が必要です。
- **高 (High):** 明らかなピーク（Q3+1.5×IQR超）。リソースの限界に近い、または異常なアクセス集中が発生している可能性があります。

### 4.2 なぜこのチャートが重要か？
- **反復パターンの発見:** 毎週同じ曜日・時間帯に負荷が集中する傾向を把握でき、予測可能な問題に対して事前に対策が立てられます。
- **異常検知の基準:** 普段は安定している時間帯に突然「警戒」や「高」が現れた場合、通常とは異なるアクセスパターンや障害の兆候を素早く察知できます。
- **リソース計画:** ピーク時間帯を正確に把握することで、スケーリングポリシーの設定やメンテナンス作業の最適な時間帯を決定できます。

### 4.3 活用ガイド
- **定期メンテナンス:** 「安定」が続く時間帯（例：日曜深夜〜月曜早朝）をメンテナンスウィンドウとして選定します。
- **オートスケーリング:** 「警戒」以上が頻出する曜日・時間帯に合わせて、サーバーの自動スケーリングルールを設定します。
- **メトリクス切替による多面的分析:** 応答時間 → エラー率 → CPU の順にメトリクスを切り替えて比較すると、ボトルネックの根本原因を特定しやすくなります。
- **期間絞り込みとの併用:** メインチャートで特定の月を選択すると、季節やイベント期間ごとの曜日パターンの変化を比較分析できます。
