# 성능 트렌드 분석 가이드

## 1. 개요
성능 트렌드 분석 대시보드는 웹 애플리케이션의 장기적인 성능 추이를 시각화하고, 시스템 상태를 다각도로 분석하기 위한 도구입니다. 1년간의 데이터를 1시간 단위의 정밀도로 유지하여, 트렌드 파악 및 용량 계획(Capacity Planning)에 활용할 수 있습니다.

## 2. 메인 시계열 차트
애플리케이션 상단에 배치된 메인 차트는 선택된 메트릭의 시간 경과에 따른 변화를 확인하는 주요 지표입니다. 1년간의 데이터를 1일 단위 정밀도로 기록하여 추세 분석과 용량 계획에 도움을 줍니다.

### 2.1 각 지표(메트릭스)의 정의
- **응답 시간 Response Time (ms):** 요청 처리 완료까지 걸린 평균 시간. 사용자 경험과 직결되는 중요한 지표입니다.
- **TPS Transactions Per Second:** 1초당 평균 처리 건수. 시스템의 현재 처리량(Throughput)을 나타냅니다.
- **Hit수 Hits (Count):** 특정 시간대 내 총 요청 횟수. 시스템의 부하량(트래픽)을 측정합니다.
- **동시 사용자 Concurrent Users:** 시스템에 활성화된 세션/사용자 수.
- **에러율 Error Rate (%):** 전체 요청 건수 대비 에러 발생 건수의 비율. 수식: `(에러 건수 / 총 요청 건수) × 100`. 에러율 상승은 애플리케이션 버그나 인프라 장애를 알리는 중요한 신호입니다.
- **CPU / 메모리:** 개별 인스턴스의 물리적 리소스 사용률. 평균값과 최대값(Peak)을 함께 분석하여 리소스 부족을 조기에 감지할 수 있습니다.

### 2.2 대시보드 조작 방법
- **기간 범위 지정:** 메인 차트 상의 특정 범위를 마우스로 드래그하면, 해당 기간의 통계 데이터가 즉시 모든 카드와 히트맵에 반영됩니다.
- **범위 이동:** 키보드 방향키 (←, →)를 활용해 선택된 1개월 구간을 앞뒤로 부드럽게 이동시킬 수 있습니다.
- **선택 해제:** `Esc` 키를 누르면 선택된 영역이 해제되며 1년 365일 전체 데이터로 되돌아갑니다.
- **이동 평균 활용:** '이동 평균' 토글을 켜면 단기적인 변동 노이즈가 제거된 장기 트렌드가 시각화되어 본질적인 추세를 쉽게 파악할 수 있습니다.

### 2.3 왜 이 차트가 중요한가?
- **장기 트렌드 시각화:** 1년 단위의 데이터를 일별로 보여줌으로써, 월간/분기별 성능 추이와 계절성 트래픽 변동을 한눈에 파악할 수 있습니다. 일시적인 스파이크와 구조적인 성능 저하를 명확하게 구분할 수 있습니다.
- **용량 계획 (Capacity Planning):** 과거의 추세 데이터를 바탕으로 미래의 리소스 수요를 예측해 "언제쯤 인프라 증설이 필요한가"를 정량적으로 판단하는 기초 자료가 됩니다.
- **변경 영향도 검증:** 배포나 인프라 변경 전후에 성능이 어떻게 달라졌는지 시계열로 확인 가능하여, 업데이트의 효과나 회귀 버그(Regression)를 즉시 감지할 수 있습니다.

### 2.4 이동 평균선 활용법
이동 평균(Moving Average)은 일별 변동 폭을 줄여 본질적인 진행 방향을 강조합니다. 본 시스템은 **7일 이동 평균**과 **30일 이동 평균** 2가지를 지원합니다.

#### ▶ 7일 이동 평균 (단기 트렌드)
- 주 단위의 경향성 변화를 빠르게 포착합니다.
- 배포 직후 며칠 내 성능에 어떤 변화가 있었는지 점검하는 데 유용합니다.

#### ▶ 30일 이동 평균 (장기 트렌드)
- 월 단위의 근본적/구조적 트렌드를 확인합니다.
- 30일선이 지속해서 우상향(응답 시간, 에러율 기준)한다면, 스케일아웃이나 최적화 검토가 필요합니다.

#### ▶ 분석 포인트
- **7일선과 30일선의 이격:** 7일선이 30일선 위로 상승하기 시작하면 최근 성능 악화의 징조이며, 아래로 내려가기 시작하면 개선을 뜻합니다.
- **골든 / 데드 크로스:** 단기선(7일)이 장기선(30일)을 밑에서 위로 돌파하면(악화 지표 기준) 데드 크로스, 위에서 아래로 교차하면 성능 개선 시그널로 해석 가능합니다.
- **원본 데이터와의 차이:** 이동 평균선 범주를 크게 벗어난 원본 데이터는 정상 범위에 속하지 않는 일회성 이벤트/장애일 확률이 높습니다.

## 3. Overall 히트맵 분석 (상관 분석)
이 차트는 **부하(Hits)**와 **성능 지표** 간의 상관관계를 나타냅니다. 전체 데이터의 중앙값(Median)을 기준으로 4개 구역(Quadrant)으로 분류됩니다. 지표 중앙값은 1년 전체 주기를 기준으로, 부하 중앙값은 현재 범위 선택된 기간을 기준으로 산출됩니다.

- **좌측 상단: 저부하 × 고지표** 부하(Hits)가 낮음에도 불구하고 Y축 수치가 높은 상태.
- **우측 상단: 고부하 × 고지표** 부하(Hits)와 Y축 수치 모두 함께 높은 상태.
- **좌측 하단: 저부하 × 저지표** 부하(Hits)와 Y축 수치 모두 함께 낮은 상태.
- **우측 하단: 고부하 × 저지표** 부하(Hits)가 높음에도 불구하고 Y축 수치가 낮은 상태.

### 3.1 메트릭별 해석 가이드
Y축에 지정된 지표 성능에 따라 각 구역의 해석이 달라집니다.

#### ▶ 응답 시간 / 에러율 / CPU / 메모리 (값이 높을수록 악화)
- **좌상단 (⚠️):** 부하가 적은데 지표가 높음 → 슬로우 쿼리, 메모리 릭 등 애플리케이션 내부의 비효율 의심.
- **우상단 (🔴):** 부하도 많고 지표도 높음 → 수용 가능(Capacity) 한계 돌파 직전. 리소스 증설과 튜닝 급선무.
- **좌하단 (🟢):** 부하도 적고 지표도 낮음 → 가장 흔하게 관측되는 안정적이고 이상적인 상태.
- **우하단 (✅):** 부하가 몰렸는데 지표는 안정적 → 매우 효율적인 아키텍쳐. 인프라 여유분 충분 상태.

#### ▶ TPS (값이 높을수록 우수)
- **좌상단 (✅):** 적은 부하로 많은 처리 달성 → 매우 효율적인 로직 실행이 이뤄지고 있습니다.
- **우상단 (🟢):** 많은 부하를 빠르게 처리 → 시스템이 이상적인 최대 성능을 발휘 중입니다.
- **좌하단 (⚪):** 낮은 트래픽에 처리 건수도 적음 → 트래픽이 한산한 시간대(유휴)입니다.
- **우하단 (⚠️):** 요청 부하는 많으나 TPS가 떨어짐 → 심각한 병목 현상 발생. 처리가 원활하게 이루어지지 않음.

### 3.2 왜 이 차트가 중요한가?
- **부하와 성능 간 인과관계 추척:** 시계열 데이터만으로 "언제 느려졌는지"는 알 수 있어도 조대체 "트래픽이 많아서인지 아니면 특정 에러가 폭주한 것인지(왜 느려진건지)" 원인 파악이 힘듭니다. Overall 히트맵은 부하 수준과의 복합 분포율을 보여줌으로써 문제 진단을 단축합니다.
- **시스템 한계 지점 도출:** 점들이 점차 우상단 쪽으로 치우치기 시작한다면, 현 구조에서는 트래픽의 추가적 폭등을 제어 불가능함을 의미하므로 선제적 조치(스케일 아웃 확보)로 이어집니다.
- **최적화 체감 검증:** 튜닝이나 인프라 교체 후 전·후를 비교했을때 데이터 분포 집합이 어디로 어떻게 전이했는지를 한눈에 대조할 수 있습니다.

### 3.3 활용 팁
- **분포 편향도 검토:** 좌측 하단(안정)에 군집이 뭉쳐있다면 아주 건전한 시스템입니다. 우상단으로 넓게 파편이 퍼질수록 리스크가 높다는 분석이 됩니다.
- **기간 범위 지정과 연동 적용:** 차트에서 특정 월 또는 주의 영역을 선택 시 집중된 기간 내 인과관계 분포 변화를 도출해 계절에 따른 특성을 교차 분석할 수 있습니다.
- **지표 전환 탐색:** 유사 기간에 메트릭만 바꿈으로써, "트래픽 쇄도로 인해 응답지연은 발생했지만 에러는 터지지 않고 버텼음"이라는 종합 컨텍스트 확보가 가능합니다.
- **인스턴스 분리 조회:** 특정 서버 인스턴스로 대상을 한정 지어 분포 파편을 대조해, 일부 서버만 노후화 문제를 겪고 있지 않은지 확인할 수 있습니다.

## 4. 요일 x 시간대별 분석 (패턴 분석)
월~일요일, 그리고 0~23시 시간대 단위로 평균치를 조각내 히트맵 패턴으로 시각화한 차트입니다. 통계적 사분위수(IQR) 기준에 기반해 스테이터스를 자동으로 감지해 분할합니다.

### 4.1 상태 기준 구분
- **안정 (Stable):** 과거 누적 통계를 투사해 볼 때 표준 편차 내 안착된 가장 정상적 기동 영역 (Q3 75% 지점 이내).
- **경계 (Warning):** 평상시보다는 지표가 두드러지게 튀기 시작한(Q3 ~ Q3 + 1.5 * IQR) 구간으로 주의 깊은 모니터링 수반이 필요.
- **위험 (High):** 통계적 이상수치(Peak - Q3 + 1.5 * IQR 초과). 임계 전선에 다다랐거나 이례적인 액세스 과밀화 현상이 발발했습니다.

### 4.2 왜 이 차트가 중요한가?
- **주기적 패턴 발굴:** 매주 목요일 오후 2시처럼 뚜렷한 주기를 갖추고 과부하가 발생하는지 패턴을 포착해, 반복되는 이슈에 대해 사전에 스케줄 제어가 가능합니다.
- **이상 탐지 가이드라인 (Anomaly Detection):** 평소 매우 파란색(안정)으로 유지되던 화요일 아침 시간에 돌연 노란색·빨간색 '위험' 마커가 출현할 경우, 일반적인 사용자 트래픽에 어긋난 돌발 상황과 장애 징표를 체감할 수 있습니다.
- **자원 계획 수립:** 트래픽 꼭지점 시간의 구체적 좌표를 획득하게 되어 탄력적 서버 할당(Auto-Scaling)과 서버 패치 작업 시간을 구상하는데 활용됩니다.

### 4.3 활용 팁
- **정기 유지보수 설정:** '안정' 빈도 구역(예: 일요일 새벽 3~5시 언저리)을 추려 가장 안전하게 점검 정비(Maintenance window)를 실행할 시간표를 편성합니다.
- **오토 스케일링 준비성:** 주로 '경계' 이상 구간에 짙게 다가가는 요일과 파트 시간대에 맞춰 규칙을 오토스케일 업(Rule) 하도록 정책 수립을 보완합니다.
- **여러 지표 전환 연속 비교:** 대시보드 상단에서 응답 시간 -> 에러율 -> CPU 등의 메트릭 구분을 순환해 누르면서 교차 분석을 하여 장애 파이프라인 구간과 구체적인 원인(Bottle-Neck)을 찾아냅니다.
- **기간 범위 지정 융합 적용:** 메인차트 하위의 슬라이드 영역을 제한시키면 곧바로 연도가 아닌 이벤트 단위나 분기 단위 요일별로의 시기적 상관 분석에 이점을 가집니다.
